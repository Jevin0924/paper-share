## 摘要

本文研究的问题是自然语言时序定位（natural-language temporal grounding）：给定一段视频 $V$ 和一个自然语言查询 $q$，系统需要返回目标事件在视频中的时间区间 $[t_s, t_e]$，其中 $t_s$ 表示起始时间，$t_e$ 表示结束时间（见 PAGE 1）。论文的核心判断是：当视频长度从几十秒扩展到小时级时，系统失败的主要原因不再是“看到附近片段后识别不了事件”，而是“无法从长视频中找到相关区域”。换言之，小时级时序定位首先是一个搜索问题（search problem），其次才是局部边界定位问题（localization problem）。

为验证这一判断，作者提出 ExtremeWhenBench，一个开放的小时级自然语言时序定位基准，包含 194 个视频和 2,273 个查询，平均视频长度 75.7 分钟，最长 9 小时，GT 事件中位长度为 9 秒（见 PAGE 1、PAGE 2）。该基准强调开放形式自然语言查询（open-form natural-language queries），而非模板化查询。实验显示，所有开放 Video-LLM 在该基准上明显崩溃；相反，一个帧级 CLIP 检索基线在小时级视频上超过所有开放 Video-LLM（见 PAGE 3）。进一步的失败分类显示，Qwen3.5-9B 的失败中 85% 属于搜索失败（见 PAGE 4）。一个简单的 retrieve-then-ground 两阶段流水线将单体 Video-LLM 的 mIoU 从 0.053 提升到 0.354，恢复幅度为 6.7 倍（见 PAGE 4）。这构成了论文最重要的实证结论：长视频自然语言时序定位应当像开放域问答中的 retrieve-then-read 一样，被分解为“先检索，再定位”。

需要说明的是，本任务未提供 `figures` 中的 `markdown_path`，因此本文不嵌入任何论文图片。下文只引用论文中的 Figure/Table 编号与页码证据。论文全文中也未提供复杂的训练目标函数或损失函数；因此，关于“公式”的分析仅限于论文实际出现的数学符号、评价表达与实验变量，不伪造额外公式。

## 背景与动机

自然语言时序定位的目标是建立视频内容与语言查询之间的时间接口。用户可能会问：“这堂课什么时候第一次介绍反向传播？”或者“会议中我们在哪一分钟同意预算？”系统需要返回一个时间区间，而不是只给出文本答案或类别标签（见 PAGE 1）。这类任务是长视频检索、视频问答、会议分析、教育视频导航和自动标注系统中的基础能力。

既有时序定位研究主要建立在短视频基准上。论文在引言中提到 Charades-STA、ActivityNet Captions、QVHighlights 等传统或常用数据集，其视频长度通常从约 30 秒到几分钟不等（见 PAGE 1、PAGE 7）。在这种短视频设置下，搜索空间较小，模型只要在有限帧范围内进行识别和边界回归，就可能取得可观结果。因此，短视频基准很难区分两个不同能力：一是模型是否能识别目标事件；二是模型是否能在很长的视频中先找到候选区域。

论文指出，长视频场景下这个差异变成结构性问题。一个 9 秒事件放在 30 秒视频里，与放在 76 分钟视频里，任务难度并不只是视频长度线性增加，而是目标事件在巨大时间搜索空间中变得稀疏。Figure 1 给出了一个关键对比：ExtremeWhenBench 将约 9 秒事件放入 76 分钟视频，而 Charades-STA 的平均视频时长约 29.8 秒；在事件粒度相近的情况下，搜索空间扩大约 153 倍（见 PAGE 1）。

现有长视频资源也不能完全满足本文需求。MAD 使用电影音频描述构建小时级查询，但只发布预计算 CLIP 特征，难以用于现代 Video-LLM 的原始视频评估；Ego4D NLQ 具有小时级第一视角视频与自然语言查询，但需要 NDA 和多 TB 下载；TVG-Bench 与 TVBench 等更适合 Video-LLM 的基准仍偏向短视频或模板化查询（见 PAGE 1、PAGE 7）。因此，论文提出了三个条件：平均长度至少 30 分钟、开放形式自然语言查询、公开 URL 且兼容标准 Video-LLM 工具链（见 PAGE 1、PAGE 7）。ExtremeWhenBench 试图同时满足这三个条件。

论文动机的另一层是方法论上的。作者将小时级时序定位类比为开放域问答（open-domain QA）中的 retrieve-then-read 框架：问答系统先检索相关文档，再让阅读器生成答案；长视频定位也应先检索相关时间区域，再在短窗口内精确定位边界（见 PAGE 2、PAGE 7）。这一类比不是简单迁移，而是由实验结果支持：当模型输入被裁剪到 GT 附近短窗口时，Video-LLM 的局部定位能力仍然较强；当输入扩展到完整 76 分钟视频时，性能急剧下降（见 PAGE 4、PAGE 10）。

## 预备知识

自然语言时序定位可表示为：给定视频 $V$ 和语言查询 $q$，预测时间区间：

$$
[t_s, t_e]
$$

其中 $t_s$ 是事件起始时间，$t_e$ 是事件结束时间。这个表达在论文摘要和引言中反复出现，是任务定义的核心符号（见 PAGE 1）。人话解释是：模型不是回答“发生了什么”，而是回答“这件事在视频的哪一段发生”。

论文使用 mIoU 作为主要评价指标（mean Intersection over Union）。虽然全文没有展开 mIoU 的形式化公式，但它在 Table 1、Figure 3、Figure 4、Table 2、Table 5、Table 7 中作为主要数值指标出现（见 PAGE 3、PAGE 4、PAGE 9、PAGE 10）。在该任务中，IoU 衡量预测时间区间与 GT 时间区间的重叠程度，mIoU 则是在样本集合上的平均。论文还规定 parse failures 计为 IoU=0（见 PAGE 3），可写作：

$$
\text{parse failure} \Rightarrow \text{IoU}=0
$$

这句话的含义是：如果模型不能输出可解析的开始/结束时间戳，则该样本按完全失败处理。

论文中的另一个关键变量是输入帧数 $N$。实验对不同模型 sweep 帧数 $N$，并报告每个模型在上下文限制内的最佳 $N$ 配置（见 PAGE 3、PAGE 9）。例如，Qwen3.5-9B 在 ExtremeWhenBench 上从 $N=128$ 到 $N=2{,}048$ mIoU 单调上升，说明更多帧覆盖带来更高搜索命中率，但即便 $N=2{,}048$ 也未达到饱和（见 PAGE 3、PAGE 9）。这里 $N$ 的意义是输入到 Video-LLM 的采样帧数量。

论文还使用 Recall@$\tau$、parse-failure rates 等附加指标，但主文只说明这些指标在 Appendix C 中给出，没有展开所有细节（见 PAGE 3）。因此，本文不进一步构造不存在的指标公式。

需要强调的是，全文材料不足以支持“论文公式 ≥5 处”的严格要求：论文主要是一篇基准与实证分解论文，而不是提出新训练目标或新网络结构的模型论文。可确认的数学表达包括 $[t_s,t_e]$、$V$、$q$、$N$、mIoU、IoU=0、R@$ \tau $、窗口长度与倍数表达等（见 PAGE 1、PAGE 3、PAGE 4、PAGE 9、PAGE 10）。下文所有数学符号均只基于这些已出现表达。

## 方法详解

### 1. 基准定义：把短视频定位问题推进到小时级搜索问题

ExtremeWhenBench 的基本设计是：保持事件粒度接近短视频基准，但大幅扩大视频时间范围。论文报告，发布集包含 2,273 个问题、194 个视频，平均视频长度 75.7 分钟，最长 9.04 小时，GT 事件中位时长 9 秒（见 PAGE 2）。Charades-STA 的事件中位时长约 7.1 秒，因此二者在目标事件粒度上接近；区别在于，ExtremeWhenBench 的事件被放入了一个约 153 倍更大的搜索空间（见 PAGE 2）。

这一设计直接服务于论文假设：如果事件本身粒度相近，而视频长度显著增加，那么性能下降更可能来自搜索困难，而不是事件识别本身变得更复杂。Figure 1 的用途是展示这种搜索空间变化；读图要点是同样短的事件，在短视频中几乎“近在眼前”，在小时级视频中则需要先被检索出来；支撑的判断是“grounding no longer reduces to recognition”（见 PAGE 1）。

论文还专门说明为什么不能简单把短视频 padding 成长视频。作者报告，将 Charades-STA 30 秒 clip 用随机素材 padding 到 10 分钟和 20 分钟后，Qwen3.5-9B 的 mIoU 从原始 0.572 降到 0.247 和 0.183；并且最差 200 个失败样本中，47% 和 74.5% 的预测落在 padding 区域（见 PAGE 2）。这说明短视频查询往往过于宽泛，例如“a person walks through the doorway”可能在多个无关场景中成立。因此，构造小时级基准不能只靠拼接噪声视频，而必须确保每个查询-区间对在原始长视频中具有唯一性。

### 2. 数据来源：只使用公开长视频作为视频基底

ExtremeWhenBench 从 LVBench、MLVU 和 VideoMME 中选取视频作为原始视频来源，共 195 个视频，平均 75.7 分钟，最长 9.04 小时（见 PAGE 2）。论文明确说明只使用这些数据集的视频源，不使用其原始 QA 标注（见 PAGE 2、PAGE 7）。这样做的好处是减少已有任务格式对新基准的影响，使新标注聚焦于自然语言时序查询。

这些源数据集具有两个现实优势。第一，它们是公开可访问资源，符合论文提出的 open access 条件。第二，它们已经与 lmms-eval 或 VLMEvalKit 等评测工具生态相关联，便于现代 Video-LLM 评估（见 PAGE 1、PAGE 2、PAGE 7）。不过，论文后文也承认，这会让基准继承 LVBench、MLVU 和 VideoMME 的类型分布，主要包括长电影、纪录片和多领域长视频，对 egocentric 或 surveillance 内容的泛化尚未单独验证（见 PAGE 5）。

### 3. 七阶段标注流水线：从帧字幕到发布查询

Figure 2 展示了 ExtremeWhenBench 的七阶段构造流程。用途是说明数据如何从原始长视频逐步变成可发布的 QA pair；读图要点是 funnel 数量变化：41,139 个 P2 验证事件，经过去重、过滤和人工审查后得到 2,273 个发布样本；支撑的判断是该基准不是直接从已有字幕粗糙抽取，而经过多阶段质量控制（见 PAGE 3）。

P0 阶段使用 Qwen3-VL-8B-Instruct 以 1 fps 生成帧级描述，每一帧输出可见实体和动作，并附带绝对视频时间戳（见 PAGE 2、PAGE 7）。这一阶段产生后续 P1-P5 的文本基础。用符号表示，视频 $V$ 被离散成一系列带时间戳的帧描述；论文没有给出正式公式，因此此处仅作概念解释。

P1 阶段使用 gpt-5-mini 对 caption stream 进行事件分组，输出候选事件及粗略边界 $[t_s,t_e]$（见 PAGE 2、PAGE 8）。P2 阶段使用 gpt-5.1 对每个候选事件做视觉边界验证：在候选区间前后各加 5 秒 padding，均匀采样 64 帧，拼成 8×8 mosaic，让 VLM 判断 VERIFIED、REFINED、SPLIT 或 REJECTED（见 PAGE 2、PAGE 8）。论文报告，原始 P1 候选中约 3.3% 被拒绝；保留事件中 79.4% 为 VERIFIED，20.6% 被 REFINED（见 PAGE 2、PAGE 8）。这意味着五分之一以上的保留样本边界经过视觉修正。

P3 阶段执行同视频内去重。gpt-5-mini 读取同一视频中的所有 P2 验证事件，只有当至少三个语义方面重合时才标记为 duplicate，例如同一 actor、同一 action、同一 setting（见 PAGE 8）。该阶段将事件数从 41,139 降至 37,599，保留率 91.4%（见 PAGE 3、PAGE 8）。这一步直接服务于唯一性要求：长视频中相似事件可能重复出现，如果不去重，自然语言查询可能对应多个时间区间。

P4 阶段生成自然语言问题。gpt-5-mini 为每个去重事件生成一个 8–18 词的问题，禁止 cinematic vocabulary，例如 “scene”“cut”“establishing shot”，并要求加入显式 ordinal disambiguation，例如 “the first time” 或 “when finally”（见 PAGE 2、PAGE 8）。这一设计避免查询依赖电影制作术语，同时减少多个候选事件之间的指代歧义。

P5 阶段是视觉评分过滤，分为 pre-pass 和 main pass。pre-pass 使用 gpt-5-mini 去除明显坏样本，并施加 25 秒 GT-duration cutoff，留下 7,188 个候选（见 PAGE 8、PAGE 9）。main pass 将相邻问题按视频分组，构建覆盖这些 GT 的 8×8 frame grids，并让 gpt-5.4 按三个标准打分：VALIDITY、UNIQUENESS WITHIN GRIDS、TEMPORAL DIFFICULTY（anti-CLIP）（见 PAGE 8、PAGE 9）。最终每组保留 top 30%，且分数不低于 7，得到 2,375 个候选（见 PAGE 2、PAGE 9）。

P6 阶段是针对性人工审查。作者使用 CLIP ViT-L/14-336 的 top-10 候选窗口来标记可能存在歧义的样本：如果 GT 区间不被任何 top-10 hit 覆盖，则说明该样本可能缺少视觉证据、边界模糊或措辞有问题（见 PAGE 9）。共 164 个样本被标记，作者人工检查后丢弃 102 个，最终得到 2,273 个发布样本，覆盖 194 个视频（见 PAGE 9）。这一步是论文质量控制中最接近人工保证的环节。

### 4. 查询侧多样性：避免模板化问题掩盖真实语言难度

论文强调，长视频定位只有在查询不是模板化的情况下才有意义。为验证开放形式查询的多样性，作者使用 TTR、MATTR、Distinct-n 和 unique 4-gram stems 进行比较（见 PAGE 2、PAGE 10）。MATTR 是 moving-average type-token ratio，用于缓解样本大小差异；unique 4-gram stems 则取每个问题前四个 token，用于检测模板前缀重复。

Table 8 显示，ExtremeWhenBench 的 MATTR 为 0.780，高于 TVBench 的 0.595 和 Charades-STA 的 0.539；Distinct-4 为 0.946，也高于 TVBench 的 0.392 和 Charades-STA 的 0.705（见 PAGE 10）。更显著的是 unique 4-gram stems：本文基准为 1,578，TVBench 为 5，Charades-STA 为 707；TVBench top-5 stem coverage 为 99.9%，而本文基准为 47%（见 PAGE 10）。

| 指标 | ExtremeWhenBench | TVBench | Charades-STA |
|---|---:|---:|---:|
| TTR | 0.173 | 0.091 | 0.058 |
| MATTR (W=50) | 0.780 | 0.595 | 0.539 |
| Distinct-2 | 0.610 | 0.230 | 0.249 |
| Distinct-4 | 0.946 | 0.392 | 0.705 |
| Unique 4-gram stems | 1,578 | 5 | 707 |
| Top-5 stem coverage | 47% | 99.9% | 19% |

表格解读：该表基于论文 Table 8（见 PAGE 10）。最关键的不是 TTR 单项，而是 MATTR 与 4-gram stems 的组合证据。MATTR 说明本文查询在局部滑动窗口中词汇变化更充分；unique 4-gram stems 说明问题开头不依赖少数模板。TVBench 的 top-5 stem coverage 达 99.9%，支持论文关于其模板化程度较高的判断。ExtremeWhenBench 的 top-5 覆盖率虽然仍有 47%，但相对 TVBench 已明显降低。

### 5. 经验分解：search 与 localize 的分离

论文的核心方法论贡献不是提出新的 Video-LLM 架构，而是将小时级 grounding 分解为 search stage 与 localize stage（见 PAGE 2）。search stage 是 query-conditioned retrieval over long temporal context，即根据自然语言查询在长视频中检索候选时间区域；localize stage 是 boundary placement within a short window，即在较短候选窗口内确定起止边界（见 PAGE 2）。

这一分解可写成概念流程：

$$
(q, V) \rightarrow \text{search candidate windows} \rightarrow [t_s,t_e]
$$

该表达不是论文中的正式公式，而是对 PAGE 2 所述 search/localize decomposition 的忠实概括。其含义是：不要让一个单体 Video-LLM 同时承担小时级搜索和秒级边界定位，而是先缩小时间范围，再执行精定位。

Figure 4 是支持该分解的关键图。用途是通过 GT-centered window 控制搜索难度，同时保持局部目标事件不变；读图要点是 Qwen3.5-9B 在 30 秒窗口下 mIoU 为 0.572，接近 Charades-STA 上限，但随着窗口扩展到 full 76 分钟降至 0.053；支撑的判断是 localize 能力在短窗口内仍然存在，而性能崩溃来自搜索空间扩大（见 PAGE 4、PAGE 10）。

### 6. Retrieve-then-ground：最小两阶段系统

论文构建了一个简单的两阶段 pipeline：先用 CLIP 根据自然语言查询检索 top-K 时间窗口，再让 Qwen3.5-9B 在候选窗口联合上下文中进行 grounding，并使用 60 秒 NMS（见 PAGE 4、PAGE 9）。该方法没有训练新的检索器，也没有修改 Video-LLM；它的目的不是成为最终模型，而是检验“分解是否必要”。

Table 2 显示，在 Qwen3.5-9B 的原生帧预算 $N=768$ 下，单体 Video-LLM Grounding 在完整 76 分钟上下文中 mIoU 只有 0.053；CLIP 检索加 rule-based thresholding 达到 0.269；而 Retrieval + Video-LLM Grounding 在 6 分钟上下文中达到 0.354（见 PAGE 4）。这意味着两阶段方法相对单体 Video-LLM 提升 6.7 倍，相对 retrieval-alone 提升 1.32 倍（见 PAGE 4）。

$$
\frac{0.354}{0.053} \approx 6.7
$$

这条计算直接对应论文 Table 2 的 “6.7× over the monolithic Video-LLM”（见 PAGE 4）。人话解释是：同一个 Video-LLM 如果先由检索器帮它找到候选区域，再做定位，性能会显著高于直接喂完整长视频。

Table 6 进一步说明 top-K 宽度存在折中。CLIP top-1±2.5 min 的 recall 为 62.8，mIoU 为 0.302；top-3±1 min 的 recall 为 81.6，mIoU 为 0.354；top-10±1 min 的 recall 提升到 96.7，但 mIoU 下降到 0.295（见 PAGE 9）。这表明候选窗口过多会重新引入长上下文搜索问题。

## 实验分析

### 1. 主实验设置

论文评估了四个开放 Video-LLM：Qwen3.5-9B、InternVL3.5-8B、LLaVA-OneVision-7B、LLaVA-NextVideo-7B；三个闭源模型：GPT-5.4、Gemini-2.5-flash、Gemini-3.5-flash；以及一个帧级语言-图像检索基线 CLIP ViT-L/14-336（见 PAGE 3）。开放模型通过 vLLM 与 lmms-eval 服务，CLIP 以 1 fps 做帧级检索（见 PAGE 3）。所有系统输出 start/end timestamp pair，解析失败按 IoU=0 处理（见 PAGE 3）。

该设置的关键在于把 Video-LLM 与检索基线放在同一个自然语言时序定位任务下比较。短视频任务中，CLIP 检索通常不一定优于强 Video-LLM，因为短视频的搜索成本较小，识别和边界定位更重要；但在小时级视频中，检索能力可能成为首要瓶颈（见 PAGE 3）。

### 2. 结果一：开放 Video-LLM 在小时级视频上系统性崩溃

Table 1 是全文最核心的实验表之一。它比较了同一类自然语言时序定位任务在 Charades-STA 与 ExtremeWhenBench 上的 mIoU。Qwen3.5-9B 从 Charades-STA 的 0.579 降到本文基准的 0.110，下降 5.3 倍；InternVL3.5-8B 从 0.359 降到 0.003，下降 120 倍；LLaVA-OneVision-7B 从 0.226 降到 0.003，下降 75 倍（见 PAGE 3）。

| 模型 | Charades-STA mIoU | ExtremeWhenBench mIoU | 下降比例 |
|---|---:|---:|---:|
| Qwen3.5-9B | 0.579 | 0.110 | 5.3× |
| InternVL3.5-8B | 0.359 | 0.003 | 120× |
| LLaVA-OneVision-7B | 0.226 | 0.003 | 75× |
| LLaVA-NextVideo-7B | 0.089 | 0.001 | 89× |
| GPT-5.4 (64f) | 0.299 | 0.013 | 23× |
| Gemini-2.5-flash (1k f) | 0.308 | 0.053 | 5.8× |
| Gemini-3.5-flash (auto-fps) | 0.466 | 0.115 | 4.1× |
| CLIP ViT-L/14-336 | 0.332 | 0.269 | 1.2× |

表格解读：该表基于论文 Table 1（见 PAGE 3）。开放 Video-LLM 在短视频上仍有非平凡表现，但在小时级视频上几乎失效。CLIP 的 mIoU 从 0.332 降至 0.269，仅下降 1.2 倍，且在 ExtremeWhenBench 上超过所有开放 Video-LLM。这说明任务难点从短视频中的局部视觉理解，转向长视频中的查询条件检索。

Figure 3 进一步支持这一点。用途是展示 Qwen3.5-9B 随输入帧数 $N$ 的性能变化；读图要点是 Charades-STA 在 $N=64$ 达到峰值 0.579 后基本饱和，而 ExtremeWhenBench 从 $N=128$ 到 $N=2{,}048$ 持续上升，且仍未饱和；支撑的判断是长视频中增加帧数仍然带来搜索覆盖提升，但仅靠增加帧数无法追上检索基线（见 PAGE 3、PAGE 9）。

### 3. 结果二：帧级检索在小时级任务中超过开放 Video-LLM

论文称这一现象为 retrieval cross-over。CLIP ViT-L/14-336 在 Charades-STA 上 mIoU 为 0.332，低于 Qwen3.5-9B 的 0.579；但在 ExtremeWhenBench 上，CLIP 达到 0.269，高于 Qwen3.5-9B 的 0.110，也高于其他开放模型（见 PAGE 3）。这并不表示 CLIP 具备更强视频理解能力，而表示在长视频条件下，能否找到相关区域比能否对近邻事件做复杂识别更重要。

这一结论对系统设计有直接含义。对于长视频自动标注、事件检索和视频问答系统，不能简单把完整视频交给一个 Video-LLM 并期待其端到端定位。更可行的工程结构是先建立可扩展的时间检索索引，再把候选片段交给更强的多模态模型处理。论文的 retrieve-then-ground hybrid 正是对这一方向的最小验证（见 PAGE 4）。

### 4. 结果三：搜索-定位交叉点随模型能力移动

Figure 4 和 Table 7 是论文区分 search 与 localize 的关键证据。实验将输入裁剪为以 GT 为中心的不同长度窗口：30 秒、1 分钟、5 分钟、10 分钟、20 分钟、40 分钟和完整 76 分钟（见 PAGE 4、PAGE 10）。这种设计保持目标事件在窗口中，但逐步增加无关上下文，从而单独改变搜索难度。

| 窗口 | Qwen3.5-9B | Gemini-3.5-flash | CLIP |
|---|---:|---:|---:|
| ±15 s / 30 s | 0.572 | 0.578 | 0.534 |
| ±0.5 min / 1 min | 0.541 | 0.542 | 0.458 |
| ±2.5 min / 5 min | 0.466 | 0.402 | 0.387 |
| ±5 min / 10 min | 0.381 | 0.413 | 0.358 |
| ±10 min / 20 min | 0.265 | 0.409 | 0.331 |
| ±20 min / 40 min | 0.098 | 0.320 | 0.300 |
| Full / 76 min | 0.053 | 0.115 | 0.269 |

表格解读：该表基于论文 Table 7（见 PAGE 10）。Qwen3.5-9B 在 30 秒窗口中与 Charades-STA 表现接近，说明其局部定位能力并未消失；但窗口越长，性能越低，并在约 20 分钟附近被 CLIP 超过。Gemini-3.5-flash 更强，在所有短窗口中保持高于 CLIP，但到完整 76 分钟时也低于 CLIP。由此可见，search-localize cross-over 的位置取决于 Video-LLM 的长上下文能力。

这组结果使论文的主张更具机制解释力。如果模型在 30 秒 GT-centered window 中也失败，那么问题可能是识别或定位能力不足；但实验显示，在短窗口中模型表现良好，窗口变长后才崩溃。因此，“搜索失败”不是泛泛的失败标签，而是由受控窗口实验定位出来的主要瓶颈。

### 5. 结果四：两阶段系统恢复大部分性能缺口

Table 2 报告的 retrieve-then-ground hybrid 是论文最有系统设计价值的实验。单体 Qwen3.5-9B 在完整 76 分钟上下文中 mIoU 为 0.053；CLIP + rule-based thresholding 为 0.269；CLIP top-3 ±1 min 检索后再由 Qwen3.5-9B 在 6 分钟上下文中定位，mIoU 达到 0.354（见 PAGE 4）。该结果显示，检索阶段不仅可以替代 Video-LLM 的长上下文搜索，还可以与 Video-LLM 的局部定位能力互补。

Table 6 的 per-K ablation 进一步说明，检索 recall 越高不一定 mIoU 越高。top-10 recall 为 96.7，但因为候选窗口合计约 20 分钟，Video-LLM 又被迫在较长上下文中搜索，mIoU 反而降至 0.295（见 PAGE 9）。这表明 retrieval stage 的目标不是无限扩大候选集，而是在覆盖率和候选上下文长度之间取得平衡。

### 6. 失败分类：85% 是搜索失败

Figure 5 展示了对 100 个 Qwen3.5-9B 失败样本的分类结果。用途是将低 IoU 失败分解为 search_fail、localization_fail 和 uncertain；读图要点是 search_fail 占 85%，localization_fail 占 11%，uncertain 占 4%，而 parsing_fail 和 refusal 均为 0%；支撑的判断是主要失败模式是选错长视频窗口，而不是拒答或格式解析失败（见 PAGE 4）。

论文的分类规则是：从 Qwen3.5-9B 在 $N=2{,}048$ 下 IoU < 0.05 的样本中随机采样 100 个，用 GPT-5.4 分类，并以预测到 GT 的 ±5 分钟偏移作为 search_fail 与 localization_fail 的区分边界（见 PAGE 4、PAGE 5）。需要注意，作者也在 limitations 中承认 ±5 分钟边界是单一阈值，对 ±2 / ±10 分钟的敏感性分析留待后续（见 PAGE 5）。因此，85% 的精确数值应被视为当前协议下的结果，而不是所有阈值下不变的常数。

## 讨论

本文的主要贡献可以概括为基准贡献和机制贡献。基准贡献是 ExtremeWhenBench：它把自然语言时序定位推进到平均 75.7 分钟的开放长视频，并通过多阶段流程控制边界质量、查询唯一性和语言多样性（见 PAGE 2、PAGE 3、PAGE 8、PAGE 9、PAGE 10）。机制贡献是实证显示小时级 grounding 的瓶颈主要是 search，而不是 localize：Video-LLM 在短窗口内仍可定位，但在完整长视频中难以找到正确区域（见 PAGE 4、PAGE 10）。

该结论对长视频系统架构有明确启示。业务上如果要做长视频自动标注、事件检索或视频问答，应优先构建 query-conditioned temporal retrieval 层，而不是只增加 Video-LLM 输入帧数或上下文长度。论文显示 Qwen3.5-9B 在 $N=2{,}048$ 时仍未饱和，但 mIoU 只有 0.110，低于 CLIP 的 0.269（见 PAGE 3）。这说明扩大输入帧数有帮助，但不是解决长视频搜索问题的充分方案。

同时，论文并不否认强 Video-LLM 的价值。Figure 4 显示 Gemini-3.5-flash 在短窗口和中等窗口中明显强于 Qwen，并在完整长度之前都高于 CLIP（见 PAGE 4、PAGE 10）。这意味着更强的 Video-LLM 可以推迟 search-localize cross-over 的发生点，但在 76 分钟完整视频上仍会被简单检索基线超过。合理解释是：Video-LLM 的局部理解能力和长上下文搜索能力是不同能力，应分别优化。

与相关工作相比，本文没有提出 RGNet 那样的统一 clip retrieval and grounding 网络，而是用最小训练自由度的混合系统证明分解必要性（见 PAGE 7）。这使论文更像是一个问题重定义与评测论文：它要求研究社区从“短视频中识别事件”转向“长视频中先搜索相关区域，再定位边界”。这种定位对后续模型设计具有约束作用。

## 局限分析

作者自述的第一类局限来自数据构造流程。ExtremeWhenBench 的 P0 caption stream 依赖单一 source-VLM，即 Qwen3-VL-8B-Instruct；作者承认尚未进行 cross-VLM caption sanity check，因此无法完全排除 caption 分布偏向某一模型家族失败模式的风险（见 PAGE 5）。这对基准可靠性有影响，因为后续事件分组、问题生成和过滤均在很大程度上依赖 P0 的 caption stream。

第二类局限来自视频类型分布。论文说明，发布基准继承了 LVBench、MLVU 和 VideoMME 的 genre distribution，主要包括长电影、纪录片和多领域长视频；对于 egocentric 或 surveillance 内容的性能没有单独测量（见 PAGE 5）。这与用户给出的风险判断一致：如果业务视频是监控、第一视角操作或目标级跟踪场景，ExtremeWhenBench 的搜索-识别分解仍有启发，但 benchmark 域和标注协议是否匹配业务视频需要重新验证。

第三类局限来自语言多样性对比。论文承认，MAD 和 Ego4D NLQ 是长度更接近的长视频参考，但由于 NDA 或 license 限制，无法直接进行 per-query MATTR 或 distinct-n 对比；因此查询多样性主张主要依赖 TVBench 和 Charades-STA 这类短视频参考（见 PAGE 5、PAGE 10）。这不推翻本文的查询多样性结果，但说明“相对长视频同类数据集更开放多样”的论断证据不足。

第四类局限来自失败分类协议。论文使用单一 ±5 分钟阈值划分 search_fail 与 localization_fail，并承认 ±2 / ±10 分钟敏感性分析留待后续（见 PAGE 5）。因此，85% search failure 是非常重要的证据，但它依赖当前阈值定义、采样规模和 GPT-5.4 分类器。若换用人工多标注者、一致性统计或不同阈值，比例可能变化。

第五类局限来自检索器选择。Finding 4 只使用 CLIP ViT-L/14-336 作为检索器，作者也指出更强的 temporally-aware, language-grounded retriever 可能改变 hybrid 的操作点和 Figure 4 中 cross-over 的位置（见 PAGE 5）。这意味着论文证明了“需要检索”，但没有穷尽“什么检索器最优”。对于实际系统，CLIP baseline 只是起点，后续应评估跨帧运动特征、语义事件索引、字幕-ASR 融合检索、多粒度时间索引等方案。

代码层面也存在证据不足。论文脚注给出 GitHub 地址，并说明更多细节将发布在 GitHub（见 PAGE 2），但当前材料没有提供可读取的 README、源码或配置文件。因此本文不能给出方法到代码的文件级映射，也不能摘录代码段。结论应写作：本文未提供可确认的公开代码，而非假定仓库已发布完整实现。

## 结论

本文提出的核心结论是：小时级自然语言时序定位首先是搜索问题。ExtremeWhenBench 通过 2,273 个开放形式查询、194 个平均 75.7 分钟的视频，把目标事件放入远大于短视频基准的时间搜索空间中（见 PAGE 1、PAGE 2）。实验显示，开放 Video-LLM 在该设置下显著崩溃，而帧级 CLIP 检索基线反而超过所有开放 Video-LLM；受控窗口实验进一步表明，局部定位能力在短窗口内仍然存在，性能下降主要来自长上下文搜索失败（见 PAGE 3、PAGE 4、PAGE 10）。

论文最有实践价值的结论是 retrieve-then-ground。一个简单的 CLIP 检索加 Qwen3.5-9B 定位系统，在只使用 6 分钟 Video-LLM 上下文的情况下达到 0.354 mIoU，相比单体 Qwen3.5-9B 的 0.053 提升 6.7 倍（见 PAGE 4）。这为长视频事件检索、自动标注和视频问答系统提供了清晰架构方向：先建立可靠的时间检索层，再使用 Video-LLM 做局部语义理解与边界定位。

对于后续研究，最自然的问题不是单纯扩大 Video-LLM 上下文，而是设计更强的 temporally-aware, language-grounded retriever，并研究检索质量如何决定 hybrid grounding 的上限（见 PAGE 5）。从这个角度看，ExtremeWhenBench 的价值不只是一个新排行榜，而是把长视频 grounding 的研究焦点从“模型能不能识别事件”转移到“系统能不能先找到事件”。
